### Container Registry Changes

> **Important:** Container registry updated from 1.15.0:

- **Previous (v1.15.0)**: `csgprdeuwrepoedge.jfrog.io/tibco-platform-docker-prod`
- **Current (v1.16.0)**: `csgprduswrepoedge.jfrog.io/tibco-platform-docker-prod`

Update your image pull secrets and registry configurations accordingly.

### Breaking Changes

> **Important:** Review these breaking changes before upgrading from v1.15.0:

1. **Container Registry URL Change**:
   - Update registry URL in Kubernetes secrets
   - Recreate image pull secrets with the new registry URL

2. **VPC CNI Network Policy (replaces Calico)**:
   - If you deployed Calico for network policies in a prior version, remove it before upgrading
   - VPC CNI `enableNetworkPolicy` is now the supported approach

3. **eksctl Metrics Server Addon**:
   - As of eksctl 0.201.0, `metrics-server` is created as a default addon unless disabled
   - The workshop recipe disables default addons (`disableDefaultAddons: true`) and manages metrics-server explicitly

4. **Helm Chart Structure**:
   - Review and update custom values files for `dp-config-aws` and `tibco-cp-base`
   - New optional parameters for resource constraints

5. **Session Management**:
   - Optional idle session timeout configuration (default: 4 hours)
   - Range: 600 seconds (10 min) to 86400 seconds (24 hours)

6. **Aurora PostgreSQL Version**:
   - Aurora PostgreSQL `engineVersion` updated to `16.8`
   - `dbParameterGroupFamily` uses `aurora-postgresql16`

## Upgrade Path

### From v1.15.0 to v1.16.0

1. **Backup Current Installation**:
   ```bash
   # Backup all Helm releases
   for release in $(helm list -n cp1-ns -o json | jq -r '.[].name'); do
     helm get values $release -n cp1-ns > ${release}-values-backup.yaml
   done
   ```

2. **Update Container Registry Credentials** (if registry URL changed):
   ```bash
   # Delete old secret
   kubectl delete secret tibco-container-registry-credentials -n cp1-ns
   
   # Create new secret with updated registry
   kubectl create secret docker-registry tibco-container-registry-credentials \
     -n cp1-ns \
     --docker-server=csgprduswrepoedge.jfrog.io \
     --docker-username=${TP_CONTAINER_REGISTRY_USER} \
     --docker-password=${TP_CONTAINER_REGISTRY_PASSWORD}
   ```

3. **Update Helm Repository**:
   ```bash
   helm repo update tibco-platform
   ```

4. **Upgrade Control Plane Components**:
   ```bash
   helm upgrade tibco-cp-base tibco-platform/tibco-cp-base \
     --version 1.16.0 \
     -n cp1-ns \
     -f cp1-values.yaml
   ```

5. **Verify Upgrade**:
   ```bash
   kubectl get pods -n cp1-ns
   helm list -n cp1-ns
   kubectl get ingress -n cp1-ns
   ```

6. **Migrate from Calico to VPC CNI Network Policy** (if applicable):
   ```bash
   # Remove Calico if previously installed
   helm uninstall -n tigera-operator tigera-operator
   
   # VPC CNI network policy is enabled via EKS addon configuration
   # Verify it's active:
   kubectl get daemonset -n kube-system aws-node -o jsonpath='{.spec.template.spec.containers[*].env}' | jq '.'
   ```

### From v1.14.0 to v1.16.0

**Not Recommended**: Direct upgrade from v1.14.0 to v1.16.0 is not supported.

**Recommended Path**: Upgrade to v1.15.0 first, then upgrade to v1.16.0.

## Known Issues & Workarounds

### EKS-Specific Considerations

1. **eksctl Metrics Server Default Addon**:
   - **Issue**: eksctl 0.201.0+ enables metrics-server by default unless `disableDefaultAddons: true` is set
   - **Workaround**: Either remove `disableDefaultAddons: true` (all defaults install), or explicitly add `metrics-server` to the addons list in `eksctl-recipe.yaml`

2. **ALB Target Group Deregistration Delay**:
   - **Issue**: During rolling updates, ALB target group deregistration takes 300 seconds (default)
   - **Workaround**: Add `alb.ingress.kubernetes.io/target-group-attributes: deregistration_delay.timeout_seconds=60` annotation for faster rolling updates in non-production environments

3. **EFS Mount Performance**:
   - **Issue**: EFS may have higher latency compared to EBS for certain workloads
   - **Workaround**: Use EBS gp3 (`ebs-gp3` storage class) for EMS capability; reserve EFS for BWCE artifacts and EMS logs

4. **VPC CNI Network Policy and Calico Conflict**:
   - **Issue**: If Calico was previously installed, VPC CNI network policies may conflict
   - **Workaround**: Remove Calico completely before enabling VPC CNI network policies

5. **Crossplane Role Privilege Escalation**:
   - **Issue**: Creating the Crossplane IAM role requires permissions to create roles and attach Administrator policy
   - **Workaround**: Use a role with sufficient privileges (e.g., account Administrator) when running the upstream [`create-crossplane-role.sh`](https://github.com/TIBCOSoftware/tp-helm-charts/blob/main/docs/workshop/eks/scripts/create-crossplane-role.sh) helper

6. **RDS SSL Not Enforced by Default (CLI method)**:
   - **Issue**: RDS instances created via CLI script do not enforce SSL by default
   - **Workaround**: Manually configure SSL enforcement per [AWS documentation](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/PostgreSQL.Concepts.General.SSL.html#PostgreSQL.Concepts.General.SSL.Requiring)
   - **Alternative**: Use Crossplane claims, which enforce SSL (`rds.force_ssl=1`) by default

## Getting Started

### Quick Start for EKS

1. **Prerequisites**:
   - AWS account with appropriate IAM permissions
   - `eksctl` 0.210.0+ and AWS CLI 2.15.0+ installed
   - Helm 3.13+ and kubectl installed
   - Route 53 hosted zone and ACM wildcard certificate

2. **Setup Guides**:
   - [Control Plane + Data Plane Setup Guide](../howto/how-to-cp-and-dp-eks-setup-guide)
   - [Data Plane Only Setup Guide](../howto/how-to-dp-eks-setup-guide)
   - [Prerequisites Checklist](../howto/prerequisites-checklist-for-customer)

3. **Key Differences from AKS/ARO**:
   - Uses AWS ALB instead of Azure Application Gateway or OpenShift Routes
   - Uses Amazon EFS and EBS instead of Azure Files/Disk
   - IRSA (IAM Roles for Service Accounts) instead of Azure Managed Identity or OpenShift SCC
   - Route 53 for DNS instead of Azure DNS
   - ACM for certificates instead of Azure Key Vault
   - `eksctl` for cluster creation instead of `az aks` or `rosa`
