# Release Notes - v1.17.0

**Release Date:** May 2026  
**TIBCO Platform Control Plane Version:** 1.17.0  
**Platform:** Amazon Elastic Kubernetes Service (EKS)

## Overview

This release updates the workshop documentation to **TIBCO Platform Control Plane version 1.17.0** for Amazon EKS. Key changes include the recommended Simplified DNS structure (single base domain for all CP services) and enhanced OTEL observability configuration.

## What's New in 1.17.0

> **Source:** [Official TIBCO Platform 1.17.0 Release Notes](https://docs.tibco.com/pub/platform-cp/latest/doc/html/Default.htm#Release-Notes/new-features.htm)

### EKS Deployment Improvements

### Simplified DNS Structure (Recommended)

- **Single base domain**: Admin UI, subscription portal, and hybrid-proxy tunnel all share one domain (e.g., `aws.example.com`) instead of separate `cp1-my.xxx` and `cp1-tunnel.xxx` subdomains
- **One ACM certificate**: A single `*.${TP_BASE_DNS_DOMAIN}` wildcard cert replaces two separate certs
- **Fewer DNS records**: Specific A-records per host (`admin.xxx`, `dev.xxx`, `tunnel.xxx`) instead of wildcard records
- **Resource savings**: Hybrid-proxy can be disabled (`CP_HYBRID_CONNECTIVITY=false`) to save ~50% CPU/RAM when multi-cloud DP connectivity is not needed
- **New env vars**: `TP_BASE_DNS_DOMAIN`, `CP_ADMIN_HOST_PREFIX`, `CP_SUBSCRIPTION`, `CP_HYBRID_CONNECTIVITY`, `TP_BASE_DOMAIN_CERT_ARN`
- **New helm values**: `adminHostPrefix`, `hybridConnectivity.enabled`, `dnsDomain`/`dnsTunnelDomain` both point to base domain

Legacy multi-level DNS (`cp1-my.xxx`, `cp1-tunnel.xxx`) remains fully supported for upgrades from 1.14.x and multi-CP-instance clusters.

### Enhanced OTEL Observability

- OpenTelemetry Collector updated to version **0.140.0** (`opentelemetry-collector-contrib`)
- Improved OTEL configuration options in all provisioner charts
- Fluent-bit OTEL configuration centralized â€” individual chart overrides removed
- Prometheus updated to **v3.5.2**, Alertmanager to **v0.32.0**

### BW5CE / BWCE Provisioner V2 Job Templates

- New parallel execution job templates (`job-bt-v2.yaml`, `job-bw-adapters-v2.yaml`, `job-bw-plugins-*-v2.yaml`) for significantly faster BW5 and BWCE provisioner bootstrapping
- V2 templates split large plugin install jobs into parallel sub-jobs â€” useful when installing many BW adapters simultaneously

## EKS-Specific Considerations for v1.17.0 CP Features

### OpenSearch on EKS
- **Amazon OpenSearch Service**: The easiest option â€” create a domain in the same VPC as your EKS cluster and connect TIBCO Platform directly
- **Self-managed OpenSearch**: Deploy via the OpenSearch Operator for Kubernetes in a dedicated namespace (e.g., `elastic-system`)
- **Existing ECK deployments**: Continue to work with v1.17.0 â€” no migration required
- Apply TIBCO Platform index templates before connecting: [OpenSearch Index Templates](https://docs.tibco.com/pub/platform-cp/latest/doc/html/UserGuide/jaeger-opensearch-index-template.htm)

### Webhook Receiver â€” Security Group and NetworkPolicy Updates
- The Control Plane namespace (`cp1-ns`) must have egress access to configured webhook endpoints
- **Security Group**: Add outbound rule for HTTPS (port 443) to the webhook endpoint IP/CIDR
- **NetworkPolicy** (if restrictive): Add egress rule from `cp1-ns` to external webhook destinations
- Test connectivity from within the cluster before configuring in the UI

### BW5CE Hawk REST API â€” Port 8090
- BW5CE pods now expose port 8090 for the Hawk REST API
- **Security Group**: If you have pod-level security groups, add port 8090 ingress rule
- **NetworkPolicy** (if restrictive): Add ingress rule for port 8090 in the Data Plane namespace
- Verify with: `kubectl exec -it <bw5ce-pod> -n dp1-ns -- curl -s http://localhost:8090/commands`

### Custom Fluentbit â€” IAM and Log Routing
- Custom Fluentbit configurations can route logs to Amazon CloudWatch, OpenSearch, or Elasticsearch
- If routing to CloudWatch: ensure the EKS node IAM role (or pod IRSA role) has `logs:PutLogEvents`, `logs:CreateLogGroup`, `logs:CreateLogStream` permissions
- If routing to Amazon OpenSearch Service: ensure VPC routing and Security Group allow traffic from Fluentbit pods to OpenSearch endpoint

## Breaking Changes

> Review before upgrading from v1.16.0:

1. **Simplified DNS is the new default** in `scripts/env.sh`. If you upgrade an existing deployment with legacy DNS, uncomment the legacy DNS block in `env.sh` (Option 2) and keep using `TP_MY_DOMAIN` / `TP_TUNNEL_DOMAIN`.

2. **Helm values for adminHostPrefix**: If migrating from legacy to simplified DNS, add `global.tibco.adminHostPrefix` and `global.tibco.hybridConnectivity.enabled` to your values file.

3. **Fluent-bit OTEL config removed from sub-charts**: If you had custom `fluent-bit` OTEL overrides in sub-chart values, move them to the top-level chart configuration.

## Upgrade Path

### From v1.16.0 to v1.17.0

1. **Backup secrets** (session-keys and cporch-encryption-secret must not change):
   ```bash
   kubectl get secret session-keys -n ${CP_INSTANCE_ID}-ns -o yaml > session-keys-backup.yaml
   kubectl get secret cporch-encryption-secret -n ${CP_INSTANCE_ID}-ns -o yaml > cporch-secret-backup.yaml
   ```

2. **Update Helm repository**:
   ```bash
   helm repo update
   ```

3. **Choose DNS approach** â€” update `scripts/env.sh`:
   - To keep legacy DNS: ensure `TP_MY_DOMAIN` and `TP_TUNNEL_DOMAIN` are set (Option 2)
   - To migrate to simplified DNS: set `TP_BASE_DNS_DOMAIN`, `CP_ADMIN_HOST_PREFIX`, `CP_SUBSCRIPTION` (Option 1)

4. **Regenerate values file** following the guide for your DNS approach, then upgrade:
   ```bash
   helm upgrade --install --wait --timeout 2h \
     -n ${CP_INSTANCE_ID}-ns tibco-cp-base tibco-cp-base \
     --repo "${TP_TIBCO_HELM_CHART_REPO}" --version "1.17.0" \
     -f aws-tibco-cp-base-values.yaml
   ```

5. **Verify**:
   ```bash
   kubectl get pods -n ${CP_INSTANCE_ID}-ns
   helm list -n ${CP_INSTANCE_ID}-ns
   ```

### From v1.15.0 to v1.17.0

Upgrade to v1.16.0 first, then to v1.17.0.

### From v1.14.0 to v1.17.0

Not supported. Upgrade incrementally: v1.14.0 â†’ v1.15.0 â†’ v1.16.0 â†’ v1.17.0.

## Known Issues

1. **External DNS wildcard alias**: Route 53 alias records support wildcard only in private hosted zones in some regions. If `*.${TP_BASE_DNS_DOMAIN}` alias doesn't propagate, create explicit A-records for `admin.xxx`, `dev.xxx`, and `tunnel.xxx` using the ALB canonical name.

2. **Hybrid-proxy disabled with legacy DNS**: Setting `CP_HYBRID_CONNECTIVITY=false` is supported only with simplified DNS. For legacy DNS, hybrid-proxy should remain enabled.
